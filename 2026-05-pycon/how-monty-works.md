# How Monty Works

Monty is a sandboxed Python interpreter written in Rust. Although it uses Ruff's parser to consume Python source code, everything below the AST is bespoke: name resolution, a bytecode compiler, a stack-based VM, a paged heap with manual reference counting plus Bacon–Rajan cycle collection, and a snapshot-and-resume protocol that hands control back to the host whenever Python code reaches out to the outside world.

The public entry point is `MontyRun` in `crates/monty/src/run.rs`. `MontyRun::new` parses and compiles the code once; `run()` / `run_no_limits()` then execute it to completion, while `start()` returns a `RunProgress` enum that can be paused and resumed for external function calls, OS calls, async waits, and unresolved names.

The end-to-end pipeline is the five phases below, executed in order.

## 1. Parse the AST with Ruff

`crates/monty/src/parse.rs` calls `ruff_python_parser::parse_module` to produce an AST identical to what Ruff/Python tooling would see. Monty doesn't keep that AST around — it walks it eagerly, converting every Ruff `Stmt`/`Expr` into Monty's own `ParseNode` / `Expr` types (see `crates/monty/src/expressions.rs`). At the same time, every identifier and string literal is fed through an `InternerBuilder`, so by the end of parsing all names are `StringId` integers rather than allocated strings.

This phase is also where source-level limits and validations are applied: `MAX_NESTING_DEPTH` (200 in release, 30 in debug) guards against pathological nesting; numeric literals are checked against `INT_MAX_STR_DIGITS`; f-strings are decomposed into Monty's own `FStringPart` / `FormatSpec` representation. Anything that's a pure syntax error fails out here as a `MontyException` with a traceback derived from the original `TextRange`.

## 2. Prepare — transform the AST

`crates/monty/src/prepare.rs` runs a second pass that turns the parsed nodes into `PreparedNode`s ready for compilation. The big job here is **name resolution**: every variable reference is classified as global, local, free (captured from an enclosing scope via a closure cell), or `LocalUnassigned` (a name that's only read, never bound — typically a host-provided external function). String-keyed lookups become integer `NamespaceId` slot indices.

Per-function scope analysis is non-trivial: it does a first-pass scan to discover assigned names, validates `global`/`nonlocal` declarations against actual bindings, computes implicit captures, and decides which variables need to be stored in closure cells because a nested function captures them. The output (`PrepareResult`) contains the prepared AST, the `namespace_size` (number of slots needed for module globals), a `name_map`, and the finalised interner. A small REPL-flavoured variant (`prepare_with_existing_names`) keeps slot IDs stable across snippets so previously compiled functions still reference the right globals.

## 3. Compile to bytecode

`Compiler::compile_module` in `crates/monty/src/bytecode/compiler.rs` walks the prepared AST and emits bytecode via `CodeBuilder`. The instruction set lives in `crates/monty/src/bytecode/op.rs`; the result is a `Code` object containing the bytecode bytes, exception-handler tables, source-location entries (so tracebacks can map IPs back to lines and carets), and constants. Nested function definitions are compiled recursively into their own `Function` objects with their own `Code`; the parent module just emits a `MakeFunction` / `MakeClosure` opcode that references the function by its index in the global function table held on `Interns`.

The compiler also handles all of Python's awkward control flow: `try`/`except`/`finally` (including `return`-inside-finally, which routes through a `finally_targets` stack), `break`/`continue` (which may need to emit `ClearException` opcodes per enclosing handler), comprehensions (compiled as anonymous functions), async/await, and decorator application order. By the time compilation finishes, the `Executor` struct holds everything execution needs: `module_code`, `interns` (now including all compiled functions), `namespace_size`, and the source string (kept only for traceback rendering).

## 4. Create a heap, run the bytecode

When `MontyRun::start` (or `run`) is called, it allocates a `Heap` sized to `namespace_size` (or a learned `heap_capacity` from a previous run) and constructs a `VM`. The heap (`crates/monty/src/heap.rs`) is a paged arena of `HeapEntry` slots, each containing a `HeapData` variant (lists, dicts, strings, tuples, dataclasses, coroutines, …), a reference count, and a Bacon–Rajan cycle-collector colour. All access goes through the `HeapReader`/`HeapRead` API — a closure-scoped, lifetime-branded handle system that makes safe reads and mutations possible without `RefCell` overhead, while a reader counter prevents an entry being freed while a handle is live. `heap.rs` and `path_security.rs` are the two most safety-critical files in the codebase.

The VM (`crates/monty/src/bytecode/vm/mod.rs`) is a conventional stack-based interpreter with two stacks: an operand stack for values and a `CallFrame` stack for function activations. Each `CallFrame` owns its own instruction pointer and a slice of the operand stack for its locals; the module-level frame has no locals because module globals live in the VM's `globals` vector instead. The main run loop fetches opcodes through a `cached_frame` macro that hoists `code`/`ip` into locals for speed, then dispatches via a giant match. Reference counts are maintained eagerly: every `Value::Ref` that's popped is dropped via `drop_with_heap`, and the `defer_drop!` / `HeapGuard` machinery makes sure refcounts stay correct even across `?`, `continue`, and exception unwinds. When refcounts dropping to non-zero values create candidate cycle roots, they're enqueued for `Heap::collect_cycles` to investigate later.

## 5. Pause for the host, resume, repeat

The VM never performs I/O, network access, or any other host interaction directly. Whenever execution reaches something only the host can do, the run loop returns a `FrameExit` variant up to `run.rs`, which converts it into a `RunProgress` for the caller. The variants are: `ExternalCall` (a Python call to an undefined-at-compile-time name — typically an injected external function), `OsCall` (filesystem and similar via `OsFunction`), `MethodCall` (a dataclass method call that must run on the original host object), `ResolveFutures` (all async tasks are blocked on external futures), `NameLookup` (an unresolved global the host might be able to provide), and the terminal `Return`. Each carries a `call_id` so the host can match resolutions to calls, plus all argument values converted into `MontyObject`s.

Each non-terminal `RunProgress` variant owns a `Snapshot<T>` containing the heap, VM state, and resource tracker — fully serializable to `postcard` bytes via `dump()`/`load()`, which is what powers iterative execution and persistence. The host calls `resume(result, print)` (or `resume_pending` for async function calls that should produce an `ExternalFuture`); that wakes the VM up, pushes the result onto the operand stack (or raises an exception if the host returned one), and re-enters the run loop. The cycle continues — execute, yield to the host, resume — until the module-level frame returns and `RunProgress::Complete(value)` is delivered to the caller. Resource limits (allocations, memory, wall time) are checked throughout via the `ResourceTracker`; tripping a limit terminates execution with a `ResourceError` and the heap/refcount state should be considered poisoned.
